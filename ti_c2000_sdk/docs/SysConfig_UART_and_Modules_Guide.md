# SysConfig guide: Add UART (SCIA) to Blinky and generalize to other modules/pins

This guide shows how to add UART (SCIA) via SysConfig to a simple LED blinky project, how to print from your application, and how to reuse the same approach for other peripherals (SPI, I2C, ePWM, ADC, etc.) and pin changes. It is based on examples already present in this C2000Ware tree so you can copy-paste with confidence.

TL;DR steps
- Open your blinky .syscfg in CCS SysConfig.
- Add SCI module, instance name “mySCIA”; set baud (9600/115200); assign SCIA RX/TX pins used by your board (examples below).
- Save .syscfg. In C, after Board_init(), print using SCI_writeCharArray(mySCIA_BASE, ...).
- For other modules, follow the same pattern: add module, create an instance, assign base (e.g., SPIA/I2CA/EPWM1), assign pins, save.

2) Add UART (SCIA) using the SysConfig GUI
- Open the blinky project’s .syscfg in CCS.
- Add a module: select “SCI” (driverlib/sci.js). Create one instance.
- Instance configuration:
  - Name: mySCIA (or any name, but refer to it consistently in C).
  - Peripheral: SCIA.
  - Baud rate: 9600 or 115200.
  - Pins:
    - F28004x LaunchPad: RX = boosterpack2.43, TX = boosterpack2.44.
    - F28002x LaunchPad: RX = boosterpack1.3,  TX = boosterpack1.4.
    - If different board: use the solver’s suggested pins or consult the matching SCI lab syscfg for your device/board under training/device/<device>/communication_peripherals/lab_sci/.
- Save the file. Board_init() will now configure clocking/pinmux for SCIA automatically.

3) Add UART (SCIA) by editing .syscfg manually (CLI-friendly)
- Insert lines like these near other module imports/instances:

```js
const sci  = scripting.addModule("/driverlib/sci.js", {}, false);
const sci1 = sci.addInstance();

sci1.$name       = "mySCIA";
sci1.baudRates   = 9600;            // or 115200
sci1.sci.$assign = "SCIA";

// Example for F28004x LaunchPad
sci1.sci.sci_rxPin.$assign = "boosterpack2.43";
sci1.sci.sci_txPin.$assign = "boosterpack2.44";

// Example for F28002x LaunchPad (alternative)
// sci1.sci.sci_rxPin.$assign = "boosterpack1.3";
// sci1.sci.sci_txPin.$assign = "boosterpack1.4";
```

- Save the file; rebuild.

4) Minimal C code to print (after Board_init)
- Include driverlib.h and board.h. Ensure Board_init() was called first.

```c
#include "driverlib.h"
#include "board.h"

static void uartPrint(const char *s)
{
#ifdef mySCIA_BASE
    // Send as a blocking array write; SCI only uses the lower 8 bits of each uint16_t
    const uint16_t *p = (const uint16_t *)s;
    uint16_t len = 0;
    while (s[len] != '\0') len++;
    SCI_writeCharArray(mySCIA_BASE, p, len);
#else
    (void)s; // SCI not present in this build
#endif
}

// Call after Board_init()
uartPrint("Hello from SCIA\r\n");
```

- Notes:
  - mySCIA_BASE is generated by SysConfig and defined in board.h.
  - SCI_writeCharBlockingNonFIFO(mySCIA_BASE, (uint16_t)ch) is fine for per-character output.

5) How to choose RX/TX pins reliably
- Best source: the SCI lab SysConfig for your device/board (training/device/<device>/communication_peripherals/lab_sci/...).
- Alternatively, let the SysConfig solver suggest legal pins after assigning SCIA, then verify with the board’s user guide/schematics (ensure those pins connect to XDS110 UART if you want USB serial).
- If using external USB-UART, choose any free SCI-capable pins and wire them accordingly.

6) Generalizing the approach to other modules and pin changes
- The same pattern applies:
  1) Find a reference SysConfig for the module on your device (training/device/<device>/<category>/lab_<module>/... or driverlib examples).
  2) Add the module with scripting.addModule("/driverlib/<module>.js").
  3) Create an instance, name it meaningfully (e.g., mySPIA, myI2CA, myEPWM1).
  4) Assign the base peripheral and pins.
  5) Save and rebuild. Board_init() will call the generated init code.

- Common modules: quick patterns
  - SPI (SPIA example):
    ```js
    const spi  = scripting.addModule("/driverlib/spi.js", {}, false);
    const spi1 = spi.addInstance();
    spi1.$name       = "mySPIA";
    spi1.spi.$assign = "SPIA";
    spi1.spi.simoPin.$assign = "<pin>";
    spi1.spi.somiPin.$assign = "<pin>";
    spi1.spi.clkPin.$assign  = "<pin>";
    // Optional STE if used
    // spi1.spi.stePin.$assign  = "<pin>";
    ```
  - I2C (I2CA example):
    ```js
    const i2c  = scripting.addModule("/driverlib/i2c.js", {}, false);
    const i2c1 = i2c.addInstance();
    i2c1.$name       = "myI2CA";
    i2c1.i2c.$assign = "I2CA";
    i2c1.i2c.sclPin.$assign = "<pin>";
    i2c1.i2c.sdaPin.$assign = "<pin>";
    ```
  - ePWM (EPWM1 example):
    ```js
    const epwm  = scripting.addModule("/driverlib/epwm.js", {}, false);
    const epwm1 = epwm.addInstance();
    epwm1.$name        = "myEPWM1";
    epwm1.epwm.$assign = "EPWM1";
    // Assign output pins as needed
    epwm1.epwm.epwmAPin.$assign = "<pin>";
    // epwm1.epwm.epwmBPin.$assign = "<pin>";
    ```
  - ADC (ADCA example; SOCs often configured in the instance UI):
    ```js
    const adc  = scripting.addModule("/driverlib/adc.js", {}, false);
    const adc1 = adc.addInstance();
    adc1.$name       = "myADCA";
    adc1.adc.$assign = "ADCA";
    // Channel input pins are selected via SOC channel dropdowns in the instance
    ```

7) Validation checklist
- Build with SysConfig: no pin conflicts reported.
- Board_init() is called in main() before using the peripheral.
- For UART: open the LaunchPad’s XDS110 COM port at the selected baud (8-N-1) and verify output.
- For SPI/I2C: probe the pins (scope/logic analyzer) or interact with the target device for expected behavior.

8) Troubleshooting
- Nothing printed on UART:
  - Ensure mySCIA was added and SCIA pins are assigned; confirm mySCIA_BASE exists in board.h.
  - Ensure Board_init() runs before printing.
  - Check terminal baud and COM port; try 9600 first.
  - Avoid pin conflicts with LEDs/other modules; let the solver re-assign or adjust LED pins.
- Build errors referencing mySCIA_BASE:
  - Indicates the SCI module wasn’t added or the instance name differs; align names or guard with #ifdef mySCIA_BASE.
- Wrong pins selected:
  - Use the corresponding lab_sci syscfg for your device/board as a ground truth or consult the LaunchPad’s schematic.
